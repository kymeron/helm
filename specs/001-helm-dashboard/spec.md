# Feature Specification: HELM Personal Dashboard

**Feature Branch**: `001-helm-dashboard`
**Created**: 2026-06-19
**Last Updated**: 2026-06-20
**Status**: 已批准 + 已实现(M1–M7)
**Input**: 用户描述: 「做一个 PC 端驾驶舱看板 web 应用,名为 HELM(舵)。每天下班后有很多事想做但不知从何开始,HELM 用来记录想法(idea)、疑问(issue)、探索(exploration),以任务卡片形式呈现,包含待办/进行中/已完成状态,并提供统计指标与图表用于跟踪分析。页面设计简约极客。」

---

## 摘要

HELM 是一款**本地优先**的个人驾驶舱看板 SPA,**核心能力**(MUST)覆盖任务 CRUD、三栏看板、统计图表、活动热力图、JSON 导入导出;**可选增量**(MAY)在同 WiFi 下通过 WebSocket 实现多设备数据互通,以及在 iPad / 手机端做响应式适配。

本文档包含 5 个用户故事:US1 看板 / US2 统计 / US3 筛选 / **US4 移动端适配 / US5 局域网同步**。

---

## 用户场景与测试 *(mandatory)*

### User Story 1 - 在看板上捕获与管理任务 (优先级: P1)

作为下班后有很多想法但不知从何开始的用户,我希望能快速记录我的想法、疑问和探索,并以看板形式看到它们当前的状态,这样我可以一眼看清自己有哪些事、哪些正在做、哪些已完成。

**Why this priority**: 这是 HELM 的核心价值——把散落的想法结构化并可视化。没有这一步,后续的统计与跟踪都无从谈起。这是可独立交付的 MVP。

**Independent Test**: 用户能新建一条任务(含类型与状态),在看板上看到该任务卡片,并通过拖拽或操作改变其状态,最终在「已完成」列看到它。全程不依赖任何其他功能。

**Acceptance Scenarios**:

1. **Given** 看板为空,**When** 用户点击「新建」并填写标题、类型(idea/issue/exploration)、状态(默认待办)后保存,**Then** 该任务卡片出现在「待办」列,并显示类型标识、优先级、创建时间。
2. **Given** 「待办」列存在一张卡片,**When** 用户将其拖拽至「进行中」列,**Then** 卡片状态变为「进行中」,列计数更新,卡片更新时间被刷新。
3. **Given** 「进行中」列存在一张卡片,**When** 用户将其拖至「已完成」列,**Then** 卡片状态变为「已完成」,系统自动记录完成时间。
4. **Given** 任一卡片,**When** 用户点击编辑并修改标题/描述/类型/优先级/标签后保存,**Then** 卡片信息更新且更新时间被刷新。
5. **Given** 任一卡片,**When** 用户点击删除,**Then** 卡片从看板移除(软删除,`deletedAt` 设值),列计数更新。
6. **Given** 状态为「已完成」的卡片,**When** 用户将其拖回「进行中」或「待办」列,**Then** 状态回退成功,完成时间被清除。
7. **Given** 用户刷新或重新打开浏览器,**When** 页面加载完成,**Then** 之前记录的所有任务卡片及其状态、顺序均恢复如初(本地持久化)。
8. **Given** 看板列标题行空白处,**When** 用户双击,**Then** 直接弹出以该列状态预选的新建表单。

---

### User Story 2 - 用统计与图表跟踪进度 (优先级: P2)

作为希望对自己的投入与完成情况进行复盘的用户,我希望能看到关键指标和图表,这样我能了解自己记录了多少想法、完成了多少、趋势如何,从而调整下一步方向。

**Why this priority**: 在能记录和跟踪任务之后,统计分析是 HELM 的第二层价值——让数据反过来指引决策,呼应「舵」的意象。它依赖 US1 的数据,但本身可独立交付与演示。

**Independent Test**: 在已有若干任务(含不同类型与状态)的前提下,用户打开页面能看到指标卡片与图表,且数值与实际任务数据一致;切换时间范围后图表随之更新。

**Acceptance Scenarios**:

1. **Given** 看板上存在若干任务(覆盖三种类型与三种状态),**When** 用户查看页面顶部指标区,**Then** 显示 7 个核心指标(任务总数、待办数、进行中数、已完成数、完成率、本周新增、本周完成),数值与实际数据一致。
2. **Given** 存在多种类型的任务,**When** 用户查看「任务类型分布」图表,**Then** 以环形图展示 idea/issue/exploration 的数量占比,hover 显示具体数量。
3. **Given** 存在已完成任务且跨越多天,**When** 用户查看「完成趋势」图表,**Then** 以双线折线图展示近 30 天每日完成与新建数量,hover 显示某日详情。
4. **Given** 存在任务数据,**When** 用户查看「活动热力图」,**Then** 以 53 周 × 7 天网格展示近一年的活动强度,GitHub 风格 6 级渐变,hover 显示具体日期与计数。
5. **Given** 存在任务数据,**When** 用户切换时间范围为「本周」/「本月」/「全部」,**Then** 指标卡片与图表数据按所选范围重新计算并刷新。
6. **Given** 任务数据发生变化(新增/状态变更/删除),**When** 用户返回统计区,**Then** 指标与图表实时反映最新数据,无需手动刷新。

### User Story 3 - 用筛选 / 排序 / 搜索精炼看板 (优先级: P3)

作为任务逐渐增多的用户,我希望能按类型、优先级、标签筛选和排序看板,并能全文搜索,这样我可以聚焦当前关心的子集,避免信息过载。

**Why this priority**: 当任务量较小时看板已足够清晰;筛选排序是任务增长后的体验优化,优先级低于核心记录与统计。可独立交付。

**Independent Test**: 在有多张不同类型/优先级/标签卡片的前提下,用户选择筛选条件后看板只显示匹配的卡片,清除筛选后恢复全部;选择排序方式后卡片顺序按规则排列;输入搜索词后匹配项保留。

**Acceptance Scenarios**:

1. **Given** 看板含多种类型的卡片,**When** 用户选择类型筛选为 `idea`,**Then** 三列中仅显示 idea 类型的卡片,其余被隐藏,列计数更新为筛选后的数量。
2. **Given** 看板含不同优先级的卡片,**When** 用户选择按优先级降序排序,**Then** 每列内卡片按 high→medium→low 顺序排列。
3. **Given** 已应用筛选,**When** 用户点击「清除筛选」,**Then** 看板恢复显示全部卡片。
4. **Given** 卡片带有自定义标签,**When** 用户选择某标签进行筛选,**Then** 仅显示含该标签的卡片。
5. **Given** 同时选择类型与优先级筛选,**When** 应用筛选,**Then** 看板显示同时满足两个条件的卡片(AND 关系)。
6. **Given** 用户在搜索框输入关键词,**When** 关键词匹配标题 / 描述 / 标签任一字段,**Then** 匹配的卡片保留,不匹配的隐藏。

### User Story 4 - 移动端适配(iPad / 手机) (优先级: P4)

作为在 iPad / 手机上也想用 HELM 记录想法的用户,我希望能直接用浏览器打开 HELM 而不需要专门适配的客户端,看板 / 模态框 / 输入框都应该可用且不被遮挡,这样我可以在不同设备间无缝切换。

**Why this priority**: 这是 v1.1 新采纳的增量(原 spec 假定 PC ≥1280px)。个人场景从『一台电脑』演进到『多设备』是自然升级路径,但属于『应有更好』而非『必须』。本期已完整实现。

**Independent Test**: 在 iPad 浏览器中打开 HELM,看板能正常浏览与新建,模态框从底部弹出且提交按钮不被键盘遮挡,任务在同 WiFi 下与 PC 同步。

**Acceptance Scenarios**:

1. **Given** 用户在 iPad Safari 上打开 HELM,**When** 页面加载,**Then** 看板竖排堆叠,TopBar 紧凑布局,无横向滚动条。
2. **Given** 用户点击新建按钮,**When** 模态框弹出,**Then** 模态框从屏幕底部弹出且圆角对齐,操作栏(sticky)始终可见。
3. **Given** 模态框打开,**When** 用户点击标题输入框并开始输入,**Then** 输入框字号 ≥16px,键盘弹出后提交按钮不被遮挡。
4. **Given** 用户完成输入,**When** 按下键盘右下角「完成」键,**Then** 表单自动提交。
5. **Given** iPad Safari 版本 < 15.4,**When** 打开 HELM,**Then** 应用不报 `crypto.randomUUID is not a function`,正常初始化。
6. **Given** 平板与 PC 在同 WiFi,**When** 平板连接到 PC 的 dev server,**Then** 平板能查看 PC 上的任务并加入同步 Hub。

### User Story 5 - 局域网多设备同步 (优先级: P5)

作为在多台设备上使用 HELM 的用户,我希望在同 WiFi 下任何一台设备上的修改都能自动同步到其他设备,这样我不需要手动导出 / 导入数据。

**Why this priority**: 这是 v1.1 新采纳的增量。属于『应有更好』,但失败不阻塞核心功能(关掉 `useSync` 即可降级为纯本地)。本期已完整实现。

**Independent Test**: 两台设备同时连接同一 dev server 的 sync hub,在 A 上新建任务,B 上能在 1 秒内看到该任务;在 A 上删除任务,B 上对应任务消失。

**Acceptance Scenarios**:

1. **Given** 设备 A 与 B 都打开了 HELM,**When** A 创建新任务,**Then** B 在 1 秒内看到该任务,且本地 IndexedDB 也已写入。
2. **Given** 设备 A 与 B 都打开了 HELM,**When** A 编辑某任务标题,**Then** B 上该任务标题同步更新(LWW 按 `updatedAt` 取新者)。
3. **Given** 设备 A 与 B 都打开了 HELM,**When** A 删除某任务,**Then** B 上该任务消失(通过 `deletedAt` 墓碑传播)。
4. **Given** 设备 A 在线,设备 B 离线,**When** B 重新连接,**Then** B 收到 Hub 的最新 snapshot 并与之对齐。
5. **Given** Hub 端 dev server 重启,**When** 设备 A 重连,**Then** A 推送本地最新 state 作为新的 Hub snapshot,其他设备重连后能收到。
6. **Given** 网络断开,**When** 用户继续在 A 上操作,**Then** 应用仍能正常使用(纯本地模式),重连后自动重新同步。

## Edge Cases

- **空状态**: 看板无任何任务时,每列显示友好的空状态提示(如「还没有想法?点击新建记录第一个」),统计区指标显示 0、图表显示空态。
- **超长标题**: 任务标题超过显示宽度时以省略号截断,hover/点击展示完整内容。
- **并发状态变更**: 同一卡片在拖拽过程中被另一操作修改时,以最后释放的操作为准,不产生中间态。
- **本地存储容量**: 当本地存储接近上限时,提示用户导出数据并清理已完成任务。
- **时间范围无数据**: 切换到某时间范围(如「本周」)但该范围内无任务时,图表显示空态而非报错。
- **状态回退的完成时间**: 已完成任务被拖回进行中/待办时,完成时间必须被清除,避免统计误算。
- **刷新时的拖拽中间态**: 页面在拖拽过程中被刷新,不应产生「半状态」数据,状态以最后一次成功释放为准。
- **旧 iPad Safari 兼容性**: `crypto.randomUUID` 不可用时自动降级到 `getRandomValues`,应用正常初始化。
- **种子数据竞态**: React StrictMode 在开发模式会 double-invoke effects,`seedIfEmpty` 必须使用固定 UUID 才能幂等,否则会出现重复任务。
- **同步断连**: WebSocket 关闭后必须能优雅降级,不影响本地 CRUD;重连后必须能重新进入同步状态。
- **selector 不稳定**: zustand v5 + useSyncExternalStore 下,任何返回新数组引用的 selector 都会导致无限重渲染直至页面 unmount,必须使用 `useShallow` 包装。

## 功能需求 *(mandatory)*

### 核心层 (MUST,所有 device 都必须满足)

- **FR-001**: 系统 MUST 允许用户创建任务卡片,字段包含标题(必填)、描述、类型(idea/issue/exploration)、状态(默认待办)、优先级(默认 medium)、标签。
- **FR-002**: 系统 MUST 以三栏看板展示任务,列分别对应「待办」「进行中」「已完成」,每列显示该列任务计数。
- **FR-003**: 系统 MUST 支持通过拖拽卡片在列间移动来变更状态,释放后状态立即更新。
- **FR-004**: 系统 MUST 在状态变更为「已完成」时自动记录完成时间;回退状态时清除完成时间。
- **FR-005**: 系统 MUST 支持编辑已有任务卡片的所有可编辑字段,并自动刷新更新时间。
- **FR-006**: 系统 MUST 支持删除任务卡片;删除为软删除(`deletedAt` 设值,记录保留);`deletedAt` 非空时 UI 与统计均隐藏该任务。
- **FR-007**: 系统 MUST 将所有任务数据持久化在浏览器本地(IndexedDB),刷新或重开浏览器后数据完整恢复;包含 `deletedAt` 墓碑。
- **FR-008**: 系统 MUST 在页面顶部展示核心指标:任务总数、待办数、进行中数、已完成数、完成率、本周(近7天)新增、本周(近7天)完成。
- **FR-009**: 系统 MUST 提供任务类型分布图(环形图)、完成与新建趋势图(近30天双线折线图)、活动热力图(53 周 × 7 天网格)。
- **FR-010**: 系统 MUST 支持统计区时间范围切换(今日/本周/本月/全部),切换后指标与图表同步刷新。
- **FR-011**: 统计指标与图表 MUST 在任务数据变化时实时更新,无需手动刷新。
- **FR-012**: 系统 MUST 支持按类型(idea/issue/exploration)、优先级(high/medium/low)、标签对看板进行筛选,多条件为 AND 关系。
- **FR-013**: 系统 MUST 支持按创建时间、优先级、更新时间对看板卡片排序。
- **FR-014**: 系统 MUST 支持全文搜索(标题 / 描述 / 标签,case-insensitive)。
- **FR-015**: 系统 MUST 支持清除筛选,一键恢复显示全部任务。
- **FR-016**: 系统 MUST 提供空状态:无任务时看板与统计区均展示友好的空态而非空白。
- **FR-017**: 系统 MUST 支持数据导出为 JSON 文件,以便用户备份与迁移;数据导入为 JSON 全量替换。
- **FR-018**: 系统 MUST 采用深色主题为默认,视觉风格简约极客,类型用颜色区分(idea/issue/exploration 各一色)。
- **FR-019**: 系统 MUST 支持看板列头双击新建(预填对应状态)。
- **FR-020**: 系统 MUST 兼容 iPad Safari < 15.4(自动检测并降级 `crypto.randomUUID`)。
- **FR-021**: 系统 MUST 在软删除 30 天后通过 `purgeTombstones` 物理清理墓碑记录。

### 可选增量 (MAY,可独立关闭)

- **FR-100** *(移动端)*: 系统 MUST 在 iPad / 手机浏览器(viewport ≥ 320px)下自动切换为紧凑布局(竖排看板、底部弹出模态框、键盘安全视口)。
- **FR-101** *(移动端)*: 系统 MUST 关闭 iOS Safari 自动缩放(输入框 ≥ 16px、`-webkit-text-size-adjust: 100%`)。
- **FR-102** *(移动端)*: 系统 MUST 在软键盘弹出时保持模态框操作栏可见(底部 sticky)。
- **FR-200** *(LAN 同步)*: 系统 MUST 在开发态下暴露 WebSocket 端点,允许多个浏览器实例连接并互相同步任务数据。
- **FR-201** *(LAN 同步)*: 同步冲突合并 MUST 使用 LWW(Last-Write-Wins)策略,以 `updatedAt` 为版本向量。
- **FR-202** *(LAN 同步)*: 删除事件 MUST 通过 `deletedAt` 墓碑字段跨设备传播;接收端 MUST 在本地反映删除。
- **FR-203** *(LAN 同步)*: WebSocket 断连 MUST 自动重连(指数退避),失败不阻塞本地 CRUD。
- **FR-204** *(LAN 同步)*: 同步层 MUST 显示状态徽章(已连接 / 连接中 / 重连中 / 出错)便于用户感知。

## 关键实体 *(include if feature involves data)*

- **Task(任务)**: HELM 的核心实体。代表一条想法/疑问/探索。关键属性:标题、描述、类型(idea/issue/exploration)、状态(todo/in_progress/done)、优先级、标签集合、创建时间、更新时间、完成时间、**删除时间(墓碑)**。状态可在三种值间流转并允许回退。
- **Tag(标签)**: 任务的自由分类维度,由用户在任务上自由填写,无独立管理实体(v1)。用于筛选与搜索。
- **SyncSnapshot(同步快照,可选增量)**: 局域网同步中的传输单元,为完整 `Task[]`。带版本信息隐式通过 `updatedAt` 表达。

## 成功标准 *(mandatory)*

### 可衡量的成果

- **SC-001**: 用户能在 30 秒内完成一次「新建任务并出现在看板」的全流程。
- **SC-002**: 拖拽卡片变更状态的操作在 200ms 内得到视觉反馈,全程保持 60fps 流畅度。
- **SC-003**: 刷新浏览器后 100% 的任务数据(含状态、字段、顺序)完整恢复,无丢失。
- **SC-004**: 统计指标与图表的数值与实际任务数据 100% 一致,可通过核对任务列表验证。
- **SC-005**: 在 200 条任务的数据量下,看板渲染与统计计算均在 1 秒内完成,无明显卡顿。
- **SC-006**: 用户无需任何说明即可理解看板三列含义与卡片状态流转(通过 5 名用户测试,≥4 人首次使用即能正确完成「记录→拖拽→完成」流程)。
- **SC-007**: 页面首屏内容在 2 秒内可见(PC 端现代浏览器、本地数据场景)。
- **SC-008** *(移动端)*: 在 iPad Safari 上完成新建任务的全流程不超过 30 秒,提交按钮在键盘弹出后仍可见可点。
- **SC-009** *(LAN 同步)*: 设备 A 上的变更在 1 秒内出现在设备 B 上(同 WiFi、≤ 50 条任务)。

## 假设

- 目标用户为单人(应用所有者),v1 不涉及多用户协作与权限。
- 用户使用现代 evergreen 浏览器(Chrome/Edge/Safari 最新版)访问,不支持 IE;**扩展兼容**: iPad Safari 15.4 以下版本。
- 桌面最小屏幕宽度 1280px;**扩展**: 移动端 ≥ 320px(iPhone SE),平板与手机做对应响应式压缩。
- v1 数据存储于浏览器本地;**扩展**: 同 WiFi 下可选启用 WebSocket 同步;广域网同步与鉴权不在 v1 范围。
- 任务量级在个人使用范围内(≤ 1000 条),统计计算无需考虑大规模性能优化。
- 标签为自由文本,v1 不提供独立的标签管理界面,仅支持在任务上填写与按已有标签筛选。
- 「本周」按近 7 天滚动计算(非自然周),便于稳定的滚动复盘。
- 默认主题为深色;v1 不提供浅色主题切换。
- 用户能容忍删除前二次确认的轻微打断,以换取防误删的安全性。**说明**: v1 实现中删除采用单步确认(详情页内「删除」按钮),由用户主动操作触发,删除为软删除可由 LAN 同步回退。
- 同步 Hub 仅用于开发态(`npm run dev` 时由 Vite 插件内嵌);生产部署需自行托管同步服务,本期不内置。
