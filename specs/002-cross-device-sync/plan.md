# Implementation Plan: HELM Cross-Device Data Sync

**Branch**: `002-cross-device-sync` | **Date**: 2026-06-21 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `/specs/002-cross-device-sync/spec.md`

## Summary

为 Vercel 生产部署引入基于 **Vercel KV + Vercel Serverless Functions** 的跨设备任务同步。客户端以匿名 token 标识用户，将完整任务快照推送到 Redis，并周期性拉取合并；本地 IndexedDB 仍是核心存储，网络不可用时自动降级为纯本地模式。

## Technical Context

**Language/Version**: TypeScript 5.x (strict mode)

**Primary Dependencies**: React 18+, Zustand, Vite, `@vercel/kv`

**Storage**: Vercel KV (Upstash Redis) for server snapshots; existing IndexedDB for local persistence

**Testing**: Vitest + React Testing Library for unit/component tests; manual validation via quickstart.md scenarios

**Target Platform**: Modern evergreen browsers (Chrome / Edge / Safari), mobile Safari included

**Project Type**: Single-page web application (SPA) with serverless API routes

**Performance Goals**: Snapshot serialization ≤ 100ms; sync pull/push ≤ 500ms; UI responsive during sync

**Constraints**: Offline-first; no long-running server; no auth friction; serverless functions only

**Scale/Scope**: Single-user, ≤1000 tasks, full snapshot model

| I. 简洁优先 | ✅ | 采用完整快照 + 单个 KV key，无增量 diff、无版本历史 |
| II. 本地优先与离线可用 | ⚠️ 需关注 | 新增服务端依赖，但本地 IndexedDB 仍是核心；网络失败时必须降级 |
| III. 类型安全默认开启 | ✅ | 新接口与快照类型继续使用 TypeScript strict |
| IV. 组件驱动 UI | ✅ | 同步逻辑放入 hooks / services，UI 仅展示状态 |
| V. 核心逻辑测试先行 | ✅ | `mergeTasks` 等核心逻辑已测试，新增同步合并逻辑需补充测试 |
| VI. 设备与网络边界清晰 | ⚠️ 需关注 | 本 feature 将可选增量扩展为 Vercel 部署必备能力，需更新宪法 Out of Scope 与 v1 Scope |

**Constitutional Conflict**: 当前宪法 `Out of Scope` 明确列出「服务器侧持久化」「公开互联网同步」。本 feature 为了解决用户 Vercel 部署后跨设备不一致问题，必须引入这两者。建议将本能力从「v1 Out of Scope」移入「v1.2+ In Scope」的可选/部署增强层，并更新原则 VI 的示例说明。

**Justification**: 用户需求明确且与现有部署平台直接相关；不引入服务端持久化则无法在 Vercel 上解决跨设备数据不一致。方案保持本地优先契约（离线可用、服务端失败可降级），复杂度可控。

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| 引入服务器侧持久化（违反宪法 Out of Scope） | Vercel 部署下 PC 与手机无法共享本地 IndexedDB | 纯本地 / 局域网同步无法跨越公网与不同设备浏览器实例 |
| 引入公开互联网同步（违反宪法 Out of Scope） | 用户明确使用 Vercel 部署，设备不在同一局域网 | WebRTC P2P 穿透复杂且不稳定，不适合普通用户 |
| 新增 `@vercel/kv` 依赖 | 需要与 Vercel KV 交互 | 自建 Redis 需要额外服务器维护，违背「部署友好」 |

## Implementation Notes

1. **Token lifecycle**: On first load, read `?token=` from URL; if absent, check `localStorage`; if still absent, generate a new token and persist it. Always reflect the active token in the URL so users can share/bookmark it.
2. **Sync trigger**: Push after every local task mutation (debounced 1s); pull every 5s when idle.
3. **Merge strategy**: When a remote snapshot arrives, merge with local tasks using existing `mergeTasks` (LWW by `updatedAt`). Write merged result back to IndexedDB.
4. **Status badge**: Reuse or extend `SyncIndicator` to show cloud sync states: `idle` (no KV config), `synced`, `syncing`, `offline`, `error`.
5. **Development fallback**: If `KV_URL` is missing, `useCloudSync` remains in `idle` and does not attempt network requests, preserving `npm run dev` without Vercel CLI.
