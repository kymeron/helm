# Feature Specification: HELM Cross-Device Data Sync

**Feature Branch**: `002-cross-device-sync`

**Created**: 2026-06-21

**Status**: Draft

**Input**: User description: 「网站使用 Vercel 部署，部署之后手机端访问和 PC 端数据不一致。新建一个迭代目录解决这个问题。」

## 摘要

HELM 当前是一款本地优先的个人看板应用：任务数据保存在浏览器 IndexedDB 中，局域网 WebSocket 同步仅在开发态（`npm run dev`）下可用。当应用部署到 Vercel 后，手机与 PC 分别访问两个独立的浏览器实例，本地数据无法互通，导致用户在不同设备上看到不一致的任务状态。

本迭代的目标是为 Vercel 生产部署引入**跨设备数据同步能力**，使用户在 PC 端新建/编辑/删除的任务，能在手机端（及任意其他设备）实时或近实时地保持一致，同时不破坏现有本地优先、离线可用的核心体验。

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 任意设备上的变更同步到其他设备 (优先级: P1)

作为在 PC 和手机上都会使用 HELM 的用户，我希望在任何一台设备上新增、编辑或删除任务后，其他设备能在合理时间内看到同样的数据，这样我就不需要手动导出/导入数据。

**Why this priority**: 这是本迭代的核心价值——解决 Vercel 部署后数据不一致的痛点。没有这一步，跨设备使用体验将不可接受。

**Independent Test**: 用户在 PC 浏览器中创建一条任务，随后在手机浏览器中打开同一部署地址，能在 5 秒内看到该任务；反之亦然。

**Acceptance Scenarios**:

1. **Given** 用户已在 PC 端登录/识别身份并存在若干任务，**When** 用户在 PC 端新建一条任务，**Then** 在手机端刷新或自动同步后，该任务出现在手机端看板中。
2. **Given** 用户在手机端编辑某任务标题，**When** 编辑保存完成，**Then** PC 端该任务标题在同步后更新为相同内容。
3. **Given** 用户在 PC 端删除某任务，**When** 删除完成，**Then** 手机端该任务在同步后消失。
4. **Given** 用户先在 PC 端离线新建任务，**When** 手机端随后在线并 PC 端也恢复在线，**Then** 双方最终数据一致，无任务丢失或重复。

---

### User Story 2 - 离线时仍可继续使用 (优先级: P2)

作为可能在网络不稳定环境下使用 HELM 的用户，我希望即使暂时无法连接同步服务，也能正常新建、编辑、查看任务，待网络恢复后自动同步。

**Why this priority**: HELM 是本地优先应用，离线可用是其核心承诺；跨设备同步不应让应用变成必须联网才能使用。

**Independent Test**: 关闭网络后，用户仍能在本地完成所有 CRUD 操作；恢复网络后，变更自动同步到其他设备。

**Acceptance Scenarios**:

1. **Given** 设备当前无法连接同步服务，**When** 用户新建/编辑任务，**Then** 操作立即在本地生效，界面无阻塞。
2. **Given** 设备在离线期间产生若干变更，**When** 网络恢复，**Then** 应用自动将本地变更推送并拉取远程变更，最终与其他设备一致。
3. **Given** 多台设备在离线期间分别产生冲突变更（如同一任务标题被不同设备修改），**When** 同步恢复，**Then** 系统按 LWW（Last-Write-Wins，以 `updatedAt` 为准）策略合并，不丢失任何一方的数据。

---

### User Story 3 - 用户身份与设备识别 (优先级: P2)

作为同一用户拥有多台设备的用户，我希望系统能识别这些设备都属于我，而不会因为换设备或清理浏览器数据导致我的任务丢失或被误判为他人数据。

**Why this priority**: 跨设备同步的前提是区分用户与设备；没有身份标识，服务端无法知道哪些设备应共享同一份数据。

**Independent Test**: 用户在 PC 和手机端通过同一身份标识访问后，两台设备看到同一份任务数据。

**Acceptance Scenarios**:

1. **Given** 用户首次在某设备打开 HELM，**When** 系统需要建立身份时，**Then** 以无密码/低摩擦方式生成或复用用户标识（如基于浏览器存储的 token、URL token、匿名身份等）。
2. **Given** 用户已在一台设备上建立身份，**When** 在另一台设备使用相同身份标识访问，**Then** 新设备拉取到该身份下的全部任务。
3. **Given** 用户浏览器本地数据被清除，**When** 用户重新打开 HELM 并提供原身份标识，**Then** 任务从服务端恢复，不会丢失。

---

### User Story 4 - 部署与运维友好 (优先级: P3)

作为将 HELM 部署到 Vercel 的用户，我希望同步方案能与 Vercel 平台兼容，不需要额外维护一台长期运行的服务器。

**Why this priority**: 用户明确使用 Vercel 部署，方案必须适配无服务器/边缘函数架构；若需要自托管服务器，会显著增加使用门槛。

**Independent Test**: 仅通过 Vercel 控制台配置环境变量并重新部署，同步功能即可在手机与 PC 之间生效。

**Acceptance Scenarios**:

1. **Given** 用户将 HELM 部署到 Vercel，**When** 按文档配置同步所需的环境变量/服务，**Then** 部署成功且手机与 PC 数据一致。
2. **Given** Vercel 函数冷启动或边缘节点切换，**When** 用户发起同步请求，**Then** 同步仍正常完成，不依赖长连接。
3. **Given** 用户未配置同步服务，**When** 应用运行时，**Then** 优雅降级为纯本地模式，不报错、不阻塞本地使用。

---

### Edge Cases

- **无网络/同步服务不可用**: 系统 MUST 降级为纯本地模式，所有本地操作继续可用。
- **冲突合并**: 同一任务在多台设备上同时修改，系统以 `updatedAt` 取最新版本，旧版本变更进入历史或被覆盖，不造成数据损坏。
- **设备首次登录空白**: 新设备首次同步时，若服务端无数据，则使用本地数据初始化；若服务端有数据，则合并本地与服务端数据。
- **浏览器清理数据**: 清除 IndexedDB 后，用户可通过身份标识从服务端恢复任务。
- **同步服务返回错误/超时**: 应用显示可感知的同步状态（如「同步失败」），但不阻止本地操作，并自动重试。
- **大数据量**: 任务数量较多时，同步不应明显阻塞 UI；考虑增量同步或分页。
- **多 Tab 同时打开**: 同一设备多个浏览器标签页同时操作，本地状态保持一致，避免重复推送。

---

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 为 Vercel 部署场景提供跨设备同步能力，使同一用户在不同浏览器实例中看到一致的任务数据。
- **FR-002**: 系统 MUST 保持本地优先体验：离线时所有 CRUD 操作立即在本地生效，联网后自动同步。
- **FR-003**: 系统 MUST 识别用户身份，使多台设备能共享同一份任务数据集。
- **FR-004**: 系统 MUST 使用 LWW（Last-Write-Wins）策略合并冲突，以 `updatedAt` 字段作为版本依据。
- **FR-005**: 系统 MUST 在同步失败或服务不可用时优雅降级，不影响本地使用，并向用户展示同步状态。
- **FR-006**: 系统 MUST 支持首次访问的设备从服务端拉取并恢复任务数据。
- **FR-007**: 系统 MUST 支持删除操作通过 `deletedAt` 墓碑字段跨设备传播。
- **FR-008**: 系统 MUST 在不配置同步服务时保持纯本地模式运行，无报错。
- **FR-009**: 系统 SHOULD 尽量减少同步延迟，使同一用户设备间的变更在 5 秒内可见。
- **FR-010**: 系统 SHOULD 适配 Vercel 无服务器/边缘架构，避免依赖长期运行的 WebSocket 服务端。

### Key Entities *(include if feature involves data)*

- **UserIdentity（用户身份）**: 代表一个 HELM 用户。关键属性：唯一标识、创建时间。用于将多台设备关联到同一任务数据集。
- **DeviceSession（设备会话）**: 代表某台设备上的应用实例。关键属性：设备标识、所属用户身份、最后同步时间。
- **Task（任务）**: 复用 HELM 核心任务实体，新增服务端持久化副本。关键属性：id、标题、描述、状态、优先级、标签、创建/更新/完成/删除时间。
- **SyncSnapshot（同步快照）**: 代表一次同步请求/响应的数据单元，可包含完整任务列表或增量变更，带服务端时间戳用于冲突判断。

---

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户在 PC 端新建任务后，手机端在 5 秒内能看到该任务（正常网络条件下）。
- **SC-002**: 用户在离线状态下完成的所有本地操作，在网络恢复后 10 秒内同步到其他设备，无数据丢失。
- **SC-003**: 同一用户的多台设备打开 HELM 后，任务总数、状态、字段内容 100% 一致。
- **SC-004**: 未配置同步服务的部署仍能 100% 通过本地 CRUD 验收，不报错、不阻塞。
- **SC-005**: 冲突场景下，系统合并后数据与 LWW 策略预期一致，无重复任务或字段损坏。
- **SC-006**: 用户从浏览器清除本地数据后，能通过身份标识在 10 秒内恢复任务。

---

## Assumptions

- 目标用户为单人使用，v1 不涉及多用户协作与权限隔离。
- 同步服务由 Vercel 兼容的服务提供（如 Vercel KV、Postgres、Edge Config、Serverless Functions 或第三方后端即服务）。
- 用户身份采用低摩擦方案（如匿名 token、URL 参数 token 或本地生成的持久化标识），v1 不强制邮箱/密码登录。
- 任务量级在个人使用范围内（≤ 1000 条），完整快照同步在可接受性能范围内；若后续增长，可再升级为增量同步。
- 用户愿意在部署时按文档配置至少一个环境变量或服务实例以启用同步。
- 现有 IndexedDB 本地存储架构保持不变，服务端数据作为可恢复的权威副本。
